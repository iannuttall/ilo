ilo runs as a local stdio MCP server:

```sh
iloso mcp serve
```

There is no remote ilo MCP URL and no browser login to ilo. The agent process uses the same local SQLite database and operating system keychain credentials as the CLI.

Connect X once before using publishing tools:

```sh
ilo start
```

## Claude Code

```sh
claude mcp add ilo -- iloso mcp serve
```

## Codex

```sh
codex mcp add ilo -- iloso mcp serve
```

## Claude Desktop and config-based clients

```json
{
  "mcpServers": {
    "ilo": {
      "command": "iloso",
      "args": ["mcp", "serve"]
    }
  }
}
```

If the client cannot find `iloso`, use the absolute path returned by `which iloso`.

## Tools

| Tool | What it does |
| --- | --- |
| `ilo_status` | Checks the local install and connected X account. |
| `ilo_create_draft` | Saves a draft to the local database. |
| `ilo_list_drafts` | Lists local drafts and their state. |
| `ilo_schedule_draft` | Schedules a draft using an ISO time or plain language. |
| `ilo_publish_draft` | Publishes one draft after explicit confirmation. |
| `ilo_publish_post` | Publishes a new post after explicit confirmation. |
| `ilo_run_scheduler` | Publishes due drafts after explicit confirmation. |

Publishing tools require `confirm: true`. Read-only and draft-creation tools do not.

## Example prompt

```txt
Create a local draft from this idea. Do not publish it.
```

When you are ready:

```txt
Show me the draft, ask me to confirm it, then publish it to X.
```

The repository is MIT licensed, so you can add tools, providers, or your own remote deployment without depending on an ilo service.

## How the local server runs

Your MCP client launches `iloso mcp serve` as a child process and communicates over stdin and stdout. The process ends when the client closes it. It does not listen on a network port, expose an OAuth callback, or require a browser login to ilo.

Because the CLI and MCP server share the same database, a draft created by an agent appears immediately in `ilo drafts list`. You can review, edit, schedule, or publish it from the terminal without transferring it between systems.

## Permission model

The MCP client controls which agents can call ilo tools. ilo adds a second guard at the action boundary: publishing a new post, publishing a draft, and running due scheduled posts all require `confirm: true`. Creating or reading a draft does not.

Use prompts that separate preparation from publication. For example, ask an agent to create three drafts first, review them, and only then approve one exact draft id for publishing. Do not give an unattended agent standing permission to confirm every write.

## Troubleshooting clients

Run `iloso mcp serve` directly to confirm that the command exists. The process will wait silently for protocol input; stop it with Control-C. If a desktop client cannot find the command, replace `iloso` with the absolute path from `which iloso` and restart the client.

Run `ilo x status` in the same user account that launches the MCP client. Keychain credentials are user-specific, so a service, container, or remote shell may not see credentials saved in your desktop session.

If a tool call fails, check the returned error before retrying. Repeated provider writes can be unsafe when a network response is ambiguous; verify the account on X before asking an agent to publish again.
