Install ilo first. Connect a publishing account if the agent will draft for a specific profile, schedule, or publish:

```sh
npm i -g iloso
ilo start
```

Public follower research can run without this connection. The guided command lets you choose a Typefully v2 API key or direct X OAuth. Typefully imports every accessible X social set. Direct X setup explains how to create an X Native App and prints `http://127.0.0.1:8976/callback` before asking for its OAuth 2.0 Client ID.

Check what the agent will see:

```sh
ilo accounts list
```

The [publishing accounts guide](/docs/accounts) covers multiple profiles, aliases, default switching, draft binding, and the split between publishing credentials and FxTwitter research.

You can now start ilo as a local stdio MCP server:

```sh
iloso mcp serve
```

The agent process uses the same local SQLite database and operating system keychain credentials as the CLI.

## Claude Code

```sh
claude mcp add ilo -- iloso mcp serve
```

## Codex

```sh
codex mcp add ilo -- iloso mcp serve
```

Restart the client after adding ilo if it does not appear immediately. Ask the agent to check `ilo_status` before creating a draft. A connected install returns the default X username, provider, and every local publishing account.

## Claude Desktop and config-based clients

```json
{
  "mcpServers": {
    "ilo": {
      "command": "iloso",
      "args": ["mcp", "serve"]
    }
  }
}
```

If the client cannot find `iloso`, use the absolute path returned by `which iloso`.

## Install the ilo skill

The MCP server gives an agent tools. The skill teaches it which tool to use, how to work from performance evidence, and when it must stop for publishing approval.

```sh
ilo skill install
```

Install both when your client supports skills. The CLI prints the installed path so you can confirm the client will discover it.

## Tools

| Tool | What it does |
| --- | --- |
| `ilo_status` | Checks the local install, default publishing account, provider, and available accounts. |
| `ilo_list_publishing_accounts` | Lists local Typefully and direct X publishing accounts. |
| `ilo_set_default_publishing_account` | Changes the default for new drafts without moving drafts already bound to another account. |
| `ilo_sync_x_followers` | Imports public X followers through FxTwitter and saves resumable progress. |
| `ilo_x_follower_sync_status` | Reports how many profiles are searchable and whether the saved import is finished. |
| `ilo_search_x_followers` | Searches imported follower profiles, returns current matches by default, and keeps counts for current, former, and unclear matches. |
| `ilo_get_x_follower_profile` | Reads the complete stored public FxTwitter record for one imported follower. |
| `ilo_create_x_article_monitor` | Saves an X handle as a local article monitor. |
| `ilo_list_x_article_monitors` | Lists article monitors and their saved history state. |
| `ilo_set_x_article_monitor_enabled` | Pauses or resumes one article monitor. |
| `ilo_delete_x_article_monitor` | Deletes an article monitor and articles saved only by it. |
| `ilo_refresh_x_articles` | Fetches new articles and continues unfinished history imports. |
| `ilo_search_x_articles` | Searches saved article titles, previews, and full text with local FTS5. |
| `ilo_get_x_article` | Reads one complete saved article and its raw provider record. |
| `ilo_create_x_monitor` | Saves an X advanced-search query as a local inbox monitor. |
| `ilo_list_x_monitors` | Lists local monitors and their latest refresh state. |
| `ilo_set_x_monitor_enabled` | Pauses or resumes one monitor. |
| `ilo_delete_x_monitor` | Deletes one local monitor. |
| `ilo_refresh_x_inbox` | Runs active monitors and saves new matching posts locally. |
| `ilo_list_x_inbox` | Filters saved posts by state, monitor, verification, relationship, or local FTS5 query. |
| `ilo_get_x_inbox_item` | Reads one saved post, author profile, monitor match, relationship state, and raw provider record. |
| `ilo_update_x_inbox_item` | Marks a local inbox item read, unread, archived, restored, replied, or unreplied. |
| `ilo_sync_x_following` | Builds or refreshes a resumable local index of complete public profiles an account follows. |
| `ilo_x_following_sync_status` | Checks following-profile coverage, completion and its reason, age, freshness, cursor, and errors. |
| `ilo_search_x_following` | Searches saved following names, handles, bios, locations, and aliases with local FTS5. |
| `ilo_get_x_following_profile` | Reads one complete stored profile and raw provider object from a following index. |
| `ilo_create_draft` | Saves a top-level post or reply draft, with optional static images, to the local database. |
| `ilo_list_drafts` | Lists local drafts and their state. |
| `ilo_schedule_draft` | Schedules a draft using an ISO time or plain language. |
| `ilo_publish_draft` | Publishes one draft after explicit confirmation. |
| `ilo_publish_post` | Publishes a top-level post or reply after explicit confirmation. |
| `ilo_run_scheduler` | Publishes due drafts after explicit confirmation. |

Draft, schedule, and publish tools accept an optional `account` alias, handle, or ID. Publishing tools require `confirm: true`. Monitor refreshes and inbox state changes write only to the local database. They never post to X.

## Example prompt

```txt
Create a local draft from this idea. Do not publish it.
```

When you are ready:

```txt
Show me the draft, publishing account, provider, and destination. Ask me to confirm all of them, then publish it to X.
```

The repository is MIT licensed, so you can add tools, providers, or new agent workflows.

## Article monitoring

The [article monitoring guide](/docs/articles) covers the complete CLI, MCP, TypeScript, storage, and history workflow. Ask an agent to monitor a few writers, refresh them, and search the saved article text:

```txt
Monitor articles from @swyx and @karpathy. Refresh both, then find articles about browser tools. Do not publish anything.
```

Article refreshes run on demand. ilo saves each timeline cursor and resumes unfinished history on later calls. New timeline entries are hydrated from their public X status so the local FTS5 index contains the complete article body rather than its short preview.

`ilo_search_x_articles` returns compact evidence excerpts. Call `ilo_get_x_article` after choosing a result to read its full stored text and raw provider record. Neither action publishes or changes anything on X.

## Reply inbox

Ask an agent to save a focused X search, refresh it, and show the strongest matches without publishing anything:

```txt
Create a monitor called "ilo mentions" for `"ilo" OR "ilo.so" -is:retweet`. Refresh it, then show unread posts from verified authors. Do not draft or publish yet.
```

Monitors are saved X advanced-search queries. A refresh runs on demand, deduplicates posts across monitors, and stores public post, author, engagement, and raw FxTwitter evidence locally. ilo does not run monitor refreshes in the background yet.

`ilo_list_x_inbox` can filter by local state, monitor id, verified author, post or author text, `followsMe`, and `iFollow`. Relationship values are tri-state. A true value means the author was found in the relevant local import; false is only available after that import completes; `null` means ilo does not have enough follower or following data to answer.

Use `ilo_sync_x_following` for `iFollow` evidence and `ilo_sync_x_followers` on the account itself for `followsMe` evidence. Do not describe an unknown relationship as false.

After the user chooses a post, an agent can call `ilo_get_x_inbox_item`, create a reply with `ilo_create_draft` and `replyToPostId`, then stop for review. Creating the draft does not publish. A later `ilo_publish_draft` call still requires `confirm: true` for that exact draft.

## Follower research

The [audience research guide](/docs/audience-research) covers the full import, search, coverage, classification, and profile workflow shared by MCP, the CLI, and TypeScript.

Follower research uses public FxTwitter data and does not require a connected X account. Ask the agent to import a bounded number of pages first:

```txt
Index 20 pages of @adamwathan's public followers, then tell me whether the saved import is finished and how many profiles are searchable.
```

The sync is resumable. Large accounts need repeated calls. Once some profiles are indexed, an agent can search them without another network request. Ask it to include former and unclear profiles when you want evidence for every classification:

```txt
Among the searchable followers of @adamwathan, how many appear to work at Cursor, Vercel, or Sentry? Separate current, former, and unclear matches, show the bio evidence, and state any possible coverage gap.
```

After finding a match, an agent can inspect all public profile data saved during the import:

```txt
Show me the complete stored profile for @leerob from @adamwathan's follower index.
```

That lookup includes the public bio, location, website, join date, account counts, avatar, banner, verification details, and raw FxTwitter object. It does not make a fresh network request.

The search tool uses SQLite FTS5 rather than an embedding model. It works best with concrete names, handles, places, and phrases. The tool normalizes common company handles such as `@getsentry`, but it does not invent synonyms or verify claims in a profile bio.

## Following research

The [following search guide](/docs/following-search) documents the full sync, freshness, search, export, profile, CLI, MCP, and TypeScript workflow.

Following research works in the other direction from follower research. It lets an agent ask questions such as "who does this account follow that is building browser tools?" The sync saves complete available public profiles rather than only relationship IDs.

Ask the agent to build a bounded snapshot first:

```txt
Index 20 pages of the public profiles @adamwathan follows. Tell me how many
complete profiles are searchable, whether the import finished, and whether the
snapshot is fresh. Do not draft or publish anything.
```

Repeat `ilo_sync_x_following` while `sync.complete` is false. Each call resumes from the local cursor unless `restart` is true. A call made after a completed snapshot begins a fresh one. `ilo_x_following_sync_status` returns the saved cursor, unique profile count, expected following count when available, timestamps, last error, age, and 24-hour stale flag.

Once profiles are saved, search makes no network request:

```txt
Search @adamwathan's saved following profiles for "building browser tools".
Return every match with its bio or location evidence and keep the snapshot
coverage in the answer.
```

`ilo_search_x_following` searches names, handles, bios, locations, and aliases with SQLite FTS5. It returns every match when `resultLimit` is omitted. The tool does not infer synonyms, verify profile claims, or turn an unfinished snapshot into proof that a missing profile is not followed.

Call `ilo_get_x_following_profile` after choosing a result. It returns the saved bio, location, website, account counts, join date, verification, images, protected state, fetch time, raw provider object, and snapshot coverage without refetching the account.

## Reply and image inputs

`ilo_create_draft` and `ilo_publish_post` accept `replyToPostId` as a numeric X post id or URL. They also accept an `images` array with up to four local JPEG, PNG, or WebP paths and optional `altText`.

The MCP process must be able to read each image path. Scheduled drafts keep those paths rather than copying the files, so the files must stay in place until publishing. Direct X supports the optional alt text. Typefully's documented v2 draft input does not, so ilo rejects a Typefully publish that contains alt text rather than dropping it. X may still reject a reply under its API access or conversation rules.

## Keep account choice explicit

Every new draft uses the requested `account` or the current local default. Its response includes a `publishingAccount` snapshot. That account remains attached if someone changes the default later.

Ask for the account before creating work for several profiles:

```txt
List my publishing accounts. Create this as a local draft for the work account,
then show me the saved account and provider. Do not publish it.
```

An agent can call `ilo_set_default_publishing_account` for a deliberate local switch. It can also leave the default alone and pass `account` to one draft, schedule, or publish call. An attempt to publish a bound draft through a different account fails before Typefully or X receives a request.

Research account handles are separate. `ilo_search_x_followers` searches the public handle in its `handle` input. Following, article, and inbox tools use `accountHandle` as the public data namespace. These tools still use FxTwitter even when the default publishing account is connected through Typefully.

## How the local server runs

Your MCP client launches `iloso mcp serve` as a child process and communicates over stdin and stdout. The process ends when the client closes it. It does not listen on a network port, expose an OAuth callback, or require a browser login to ilo.

Because the CLI and MCP server share the same database, a draft created by an agent appears immediately in `ilo drafts list`. You can review, schedule, or publish it from the terminal without transferring it between systems.

## Permission model

The MCP client controls which agents can call ilo tools. ilo adds a second guard at the action boundary: publishing a post or reply, publishing a draft, and running due scheduled posts all require `confirm: true`. Creating or reading a draft does not.

Use prompts that separate preparation from publication. For example, ask an agent to create three drafts first, review them, and only then approve one exact draft ID for publishing. Before approval, check the publishing account, provider, text, top-level or reply destination, images, and alt text. Do not give an unattended agent standing permission to confirm every write.

## Troubleshooting clients

Run `iloso mcp serve` directly to confirm that the command exists. The process will wait silently for protocol input; stop it with Control-C. If a desktop client cannot find the command, replace `iloso` with the absolute path from `which iloso` and restart the client.

Run `ilo accounts list` in the same operating-system user account that launches the MCP client. Keychain credentials are user-specific, so a service, container, or remote shell may not see credentials saved in your desktop session.

If a tool call fails, check the returned error before retrying. Repeated provider writes can be unsafe when a network response is ambiguous; verify the account on X before asking an agent to publish again.
