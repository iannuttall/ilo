ilo is an open source CLI, local MCP server, and agent skill for drafting, scheduling, and publishing to X.

## Install the CLI

```sh
npm i -g iloso
```

Both `ilo` and `iloso` run the same command.

## Create an X app

Open the [X Developer Console](https://console.x.com) and create or select an app. Under **User authentication settings**, enable OAuth 2.0, choose **Native App**, and use read and write permissions. Native Apps are public clients, so ilo uses PKCE and does not need a client secret.

Add this callback URL:

```txt
http://127.0.0.1:8976/callback
```

The X API may have its own usage charges. Those go directly to X, not ilo.

## Connect your account

```sh
ilo start
```

The CLI shows the same setup details and callback URL before it asks for anything. Paste the OAuth 2.0 Client ID from **Keys and tokens** when prompted. ilo opens X in your browser, waits for the local callback, and saves the tokens in your operating system keychain.

You can also pass the client ID directly:

```sh
ilo x connect --client-id YOUR_CLIENT_ID
```

If you intentionally configured a confidential Web App or Automated App instead, pass its secret with `--client-secret`. The normal Native App flow does not use one.

Check the connection:

```sh
ilo x status
```

## Publish your first post

```sh
ilo post --text "Hello from ilo"
```

ilo asks for confirmation before publishing. For non-interactive scripts, pass `--yes`.

## Create and schedule a draft

```sh
ilo drafts create --text "A post for tomorrow"
ilo drafts list
ilo drafts schedule <draft-id> --at "tomorrow at 9am"
ilo scheduler watch
```

The scheduler is a local process. Keep `ilo scheduler watch` running, or call `ilo scheduler run` from cron or another task runner.

## Connect an agent

```sh
claude mcp add ilo -- iloso mcp serve
codex mcp add ilo -- iloso mcp serve
```

The MCP server uses stdio and reads the same local database and keychain credentials as the CLI. Publishing tools require an explicit `confirm: true` argument.

## Check where ilo stores data

Drafts, schedules, publish attempts, and settings stay in a local SQLite database. Account metadata lives in `~/.config/ilo/config.json`; OAuth tokens and an optional client secret are stored in your operating system keychain.

The CLI, MCP server, and agent skill use the same drafts, schedules, and publishing workflow.

## Keep the scheduler running

`ilo scheduler watch` only publishes while that process is running. For a machine that sleeps or restarts, run `ilo scheduler run` from cron, launchd, systemd, or another task runner at a regular interval. Running it more than once is safe: only drafts that are due and still scheduled are selected.

Use an explicit timezone when a schedule must be unambiguous. You can inspect drafts before publishing with:

```sh
ilo drafts list
ilo scheduler run --json
```

## Troubleshooting

If browser authorization fails, confirm that the X app callback URL exactly matches `http://127.0.0.1:8976/callback` and that OAuth 2.0 read and write access is enabled. Run `ilo x status` after reconnecting.

If an agent cannot start ilo, use the absolute executable path returned by `which iloso` in the MCP configuration. The server writes protocol messages to stdout, so keep shell banners and debug output away from the MCP process.

Disconnecting removes the saved X connection without deleting local drafts:

```sh
ilo x disconnect
```

Read the [CLI reference](/docs/cli) or [local MCP guide](/docs/mcp) for the full surface.
