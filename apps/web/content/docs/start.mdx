Install ilo, choose how it should publish, and review one post end to end. The same local setup then works from the CLI, MCP server, agent skill, and TypeScript package.

## Install the CLI

```sh
npm i -g iloso
```

Both `ilo` and `iloso` run the same command.

## Choose Typefully or direct X

Run the guided setup:

```sh
ilo start
```

Choose **Typefully API key** if your X account is already connected to Typefully. Open Typefully **Settings → API**, create a v2 API key, and paste it into ilo's hidden prompt. ilo imports every accessible Typefully social set that has an X profile and stores the key in your operating system keychain.

Choose **Direct X OAuth** when you want ilo to publish straight to X through your own developer app. Open the <a href="https://console.x.com" target="_blank" rel="noreferrer">X Developer Console</a> and create or select an app. Under **User authentication settings**, enable OAuth 2.0, choose **Native App**, and use read and write permissions.

Register this exact callback URL:

```txt
http://127.0.0.1:8976/callback
```

Native Apps are public clients, so ilo uses PKCE and does not need a client secret. Copy the OAuth 2.0 Client ID from **Keys and tokens**. X controls its developer plans, limits, and charges.

The [publishing accounts guide](/docs/accounts) compares both routes, covers multiple X profiles, and explains the current Typefully image-alt-text limit.

## Check the account ilo will use

List the connected accounts and current default:

```sh
ilo accounts list
```

A filled marker shows the default. Typefully may add several X profiles from one API key. Switch by local alias or handle:

```sh
ilo accounts use work
```

New drafts use this default unless you pass `--account`. The chosen publishing route does not change research reads. Public follower, following, article, and inbox data still comes through FxTwitter and stays in local SQLite.

## Publish your first reviewed post

Start with a local draft so you can inspect the account binding:

```sh
ilo drafts create --text "Hello from ilo"
ilo drafts list
```

The list includes the X handle and provider attached to the draft. Publish it when both are correct:

```sh
ilo drafts publish <draft-id>
```

ilo shows the publishing account, provider, destination, text, and images before it asks for confirmation. Nothing is sent before you accept. Use `--yes` only when a separate caller already recorded approval for that exact action.

You can publish one reviewed post without keeping a draft first:

```sh
ilo post --account work --text "Hello from ilo"
```

## Create and schedule a draft

```sh
ilo drafts create --text "A post for tomorrow"
ilo drafts list
ilo drafts schedule <draft-id> --at "tomorrow at 9am"
ilo scheduler watch
```

The scheduler is a local process. Keep `ilo scheduler watch` running, or call `ilo scheduler run` from cron or another task runner. Typefully schedules also stay local in this release; ilo sends the post to Typefully when it becomes due.

## Connect an agent

```sh
claude mcp add ilo -- iloso mcp serve
codex mcp add ilo -- iloso mcp serve
```

The MCP server uses stdio and reads the same local database and keychain credentials as the CLI. Publishing tools require an explicit `confirm: true` argument.

## Check where ilo stores data

Drafts, schedules, publish attempts, and settings stay in a local SQLite database. Account metadata lives in `~/.config/ilo/config.json`. Typefully API keys, X OAuth tokens, and an optional X client secret are stored in your operating system keychain.

The CLI, MCP server, and agent skill use the same drafts, schedules, and publishing workflow.

## Keep the scheduler running

`ilo scheduler watch` only publishes while that process is running. For a machine that sleeps or restarts, run `ilo scheduler run` from cron, launchd, systemd, or another task runner at a regular interval. Running it more than once is safe: only drafts that are due and still scheduled are selected.

Use an explicit timezone when a schedule must be unambiguous. You can inspect drafts before publishing with:

```sh
ilo drafts list
ilo scheduler run --json
```

## Troubleshooting

If direct X browser authorization fails, confirm that the callback exactly matches `http://127.0.0.1:8976/callback` and that OAuth 2.0 read and write access is enabled. Run `ilo x status` after reconnecting.

If Typefully setup fails, create a current v2 key under **Settings → API** and confirm the Typefully user can write to at least one social set with an X profile. `ilo accounts list` shows what was saved locally.

If an agent cannot start ilo, use the absolute executable path returned by `which iloso` in the MCP configuration. The server writes protocol messages to stdout, so keep shell banners and debug output away from the MCP process.

Remove one publishing connection without deleting published draft history:

```sh
ilo accounts remove <alias-or-handle>
```

Accounts with scheduled drafts are protected from removal until those drafts publish. Unscheduled drafts become unassigned and can be bound to another account later.
