Install ilo from npm:

```sh
npm i -g iloso
```

Run `ilo --help` to see the current command tree.

## Connect one or more publishing accounts

```sh
ilo start
ilo accounts add typefully
ilo accounts add x --alias work
ilo accounts list
ilo accounts use work
ilo accounts remove <alias-or-handle>
```

`ilo start` asks whether you want to publish through Typefully or directly through your own X developer app. Typefully is the quicker route when your X profiles already live there. A direct X connection is useful when you want to avoid another publishing service or need image alt text.

Typefully setup asks for a v2 API key from **Settings → API**. It adds every accessible Typefully social set with an X profile. Direct X setup shows the X developer portal, recommends a Native App with OAuth 2.0 read and write access, and prints the exact callback before asking for a Client ID:

```txt
http://127.0.0.1:8976/callback
```

Copy the OAuth 2.0 Client ID from the app's **Keys and tokens** page. A Native App uses PKCE and does not need a client secret. `ilo x connect`, `ilo x status`, and `ilo x disconnect` remain available for direct X accounts.

Account aliases and non-secret metadata live in `~/.config/ilo/config.json`. Typefully keys, X OAuth tokens, and optional X client secrets are stored in the operating system keychain.

New drafts use the current default account. Pass `--account <alias-or-handle>` to target another account for one command. A draft remembers the account it was created for, so switching the default does not move older drafts to a different X profile.

The [publishing accounts guide](/docs/accounts) covers provider choice, duplicate handles, draft binding, local scheduling, account removal, and how publishing accounts relate to research handles.

## Publish one post

```sh
ilo post --text "Post text"
ilo post --account work --text "Post text for work"
ilo post --file ./post.txt
printf "Post from stdin" | ilo post
```

Interactive publishing shows the exact handle, provider, text, destination, and media before asking for confirmation. Use `--yes` only when the caller has already confirmed that exact post and publishing account.

## Reply and attach images

Pass an X post id or URL to publish a reply:

```sh
ilo post --reply-to https://x.com/example/status/123 --text "Reply text"
```

Attach up to four JPEG, PNG, or WebP files. Repeat `--image` and pair each file with `--alt` in the same order:

```sh
ilo post \
  --text "Weekly signups" \
  --image ./chart.png \
  --alt "Bar chart of weekly signups by source"
```

The same options work with `ilo drafts create`, including scheduled drafts. ilo saves absolute image paths with the draft, so the files must remain available until it publishes. Each image can be up to 5 MB.

Both publishing routes accept static images. Typefully's documented v2 draft input does not expose image alt text, so ilo rejects a Typefully post that includes `--alt` instead of silently deleting the text. Use a direct X publishing account for that post.

X applies its own API access and conversation rules to replies. A reply that works in the X website can still be rejected by the API.

## Search public followers

The [audience research guide](/docs/audience-research) walks through import coverage, classifications, freshness, complete profile records, and exports across the CLI, MCP, and TypeScript package.

Import public follower profiles. The default command reads 20 pages and saves progress, so rerunning it resumes a large import:

```sh
ilo x followers sync adamwathan --pages 20
ilo x followers status adamwathan
```

Use `--all` to continue until ilo reaches the end of the available list, or until the saved profiles match X's reported total and repeated pages add nothing new:

```sh
ilo x followers sync adamwathan --all
```

For a large account, run the same full import as a detached local process:

```sh
ilo x followers sync adamwathan --background
ilo x followers status adamwathan
```

The start command prints the progress log path. The status command shows the background PID while it is running. Every page is committed to local SQLite, so a stopped job can resume without starting over.

Read the complete public profile record stored for one imported follower:

```sh
ilo x followers profile adamwathan leerob
ilo x followers profile adamwathan leerob --json
```

The JSON result includes the public bio, location, website, join date, account counts, avatar, banner, verification details, and complete stored public record. Older rows imported before these fields were added need a `--restart` sync to fill them in.

Search the local names, handles, bios, and locations with SQLite FTS5:

```sh
ilo x followers search adamwathan \
  --query "how many of my followers work at cursor|vercel|sentry"
```

Employer searches keep `current`, `former`, and `unclear` matches separate and return the public bio used as evidence. Current matches are listed by default while the counts still include all three classifications. Check the coverage lines first. If the saved import is unfinished, ilo shows the reported follower total and an approximate possible gap. A public bio is not a verified employment record.

Add either flag when you want to inspect the noisier classifications:

```sh
ilo x followers search adamwathan \
  --query "works at cursor|vercel|sentry" \
  --include-former \
  --include-unclear
```

Matching profiles use a wide table when it fits and stacked records in a normal or narrow terminal. Match counts appear below the profiles. Every profile selected by the classification flags is shown unless you pass `--limit`:

```sh
ilo x followers search adamwathan \
  --query "works at cursor|vercel|sentry" \
  --limit 10
```

Export the matching public profiles when you want to inspect them in a spreadsheet or make a chart:

```sh
ilo x followers search adamwathan \
  --query "works at cursor|vercel|sentry" \
  --csv ./adam-company-followers.csv
```

The CSV has one row per company and listed profile match. It includes the classification, bio evidence, account counts, location, website, verification details, public image URLs, join date, and import coverage. Current matches are exported by default. Add `--include-former` or `--include-unclear` to export those profiles too. Search classifies up to 10,000 candidates for each company by default. Change that cap with `--candidates`.

## Search the profiles an account follows

The [following search guide](/docs/following-search) covers complete profile storage, resumable syncs, automatic snapshot refreshes, freshness, local search, exports, MCP, TypeScript, and evidence limits.

Build the default publishing handle's profile index by omitting the handle, or pass a public handle explicitly:

```sh
ilo x following sync --all
ilo x following sync adamwathan --all
ilo x following status adamwathan
```

Without `--all`, sync reads 20 pages by default and saves its cursor. Run it again to continue, or pass `--pages 100` to choose a different budget. An unfinished import resumes. Running sync after a completed import starts a fresh membership snapshot. Use `--restart` only when you want to discard an unfinished cursor.

Search the saved names, handles, bios, locations, and aliases with local SQLite FTS5:

```sh
ilo x following search adamwathan \
  --query "building browser tools"
```

The search command never starts a sync. If no complete profiles are indexed, it tells you which sync command to run. Search returns every local match unless you pass `--limit`, and the output keeps the exact total visible when a display limit is active.

Export every match and its import coverage to CSV:

```sh
ilo x following search adamwathan \
  --query "building browser tools" \
  --csv ./adam-browser-builders.csv
```

CSV export ignores a separate terminal `--limit` so a display choice cannot silently truncate the file. Use `--json` for structured results with `totalMatches`, `truncated`, complete profile fields, and coverage.

Inspect one stored profile without making another network request:

```sh
ilo x following profile browser_builder --account adamwathan
ilo x following profile browser_builder --account adamwathan --json
```

Status and search report whether ilo reached the end of the available list or repeated pages confirmed X's reported total. They also report whether the snapshot is more than 24 hours old. A partial result can prove which profiles were found, but not that an absent profile is not followed.

## Monitor X articles

Add the handles whose articles you want to keep, then refresh them when you want new writing:

```sh
ilo x articles add swyx
ilo x articles add karpathy
ilo x articles refresh
```

ilo reads each handle's public article timeline and fetches the complete body for every new article. The timeline cursor and article text stay in local SQLite. Refreshes do not run in the background and never publish anything.

Each refresh reads up to three pages per monitor. When older articles remain, ilo prints the command to run again and resumes from the saved cursor.

Search saved titles, previews, and full article text with SQLite FTS5:

```sh
ilo x articles search --query "browser tools"
ilo x articles search --from swyx
ilo x articles search --query "agents" --limit 10 --json
```

Search returns every match unless you pass `--limit`. Normal output shows an evidence excerpt and the X link. JSON search results stay compact too. Read the full stored body and complete public record when you choose an article:

```sh
ilo x articles show <post-id-or-url>
ilo x articles show <post-id-or-url> --json
```

The [article monitoring guide](/docs/articles) covers history state, every monitor command, JSON fields, local search behaviour, and current limits.

## Build a local reply inbox

Save an X advanced-search query as a monitor, then refresh it when you want new matches:

```sh
ilo x monitors add "ilo mentions" \
  --query '"ilo" OR "ilo.so" -is:retweet'
ilo x inbox refresh
ilo x inbox list
```

A refresh saves matching public posts, author profiles, engagement counts, the matching monitor, and the complete public post record to local SQLite. It deduplicates posts found by more than one monitor. Monitors do not run in the background yet, so rerun `ilo x inbox refresh` when you want to check them.

Filter the saved inbox by monitor, local state, verified authors, or words in the post and author profile:

```sh
ilo x inbox list --monitor "ilo mentions" --unread
ilo x inbox list --verified --query "follower search"
ilo x inbox archive <post-id-or-url>
ilo x inbox restore <post-id-or-url>
```

Sort by signal when you want useful candidates ahead of the newest match. `--explain` prints the strongest positive and negative factors beside each item:

```sh
ilo x inbox list --language en --sort signal --explain
```

The score compares visible response with views or account size when those fields exist. It also uses freshness, specific evidence, public author fields, known relationships, promotional wording, near-duplicates, and local feedback. Verification adds little on its own. The score is an ordering aid with a confidence label, not a probability that a post is good or an author is trustworthy.

Teach the local ranking after you review an item. Reasons make the preference easier to inspect later:

```sh
ilo x inbox useful <post-id-or-url> --reason actionable
ilo x inbox dismiss <post-id-or-url> --reason promotional
ilo x inbox feedback-clear <post-id-or-url>
```

Useful and dismissed choices stay in local SQLite. They affect the selected post directly and give later posts from the same author a cautious adjustment. Clearing feedback removes that choice.

Import the default publishing handle's following profiles to use `--i-follow`. The same index is searchable with `ilo x following search`. The existing follower import powers `--follows-me`:

```sh
ilo x following sync --all
ilo x followers sync <your-handle> --all
ilo x inbox list --i-follow
ilo x inbox list --follows-me
```

Relationship fields can be `true`, `false`, or `unknown`. A known match can be true during a partial import, but ilo only reports false after the relevant full import completes. Without that evidence, the terminal says unknown and JSON returns `null`.

Inspect one item or turn it into a local reply draft:

```sh
ilo x inbox show <post-id-or-url>
ilo x inbox draft <post-id-or-url> --text "Reply to review"
```

Drafting marks the inbox item as read. It sets the reply target on a local draft and prints `Nothing was published.` Publishing that draft still requires the normal confirmation step. A successful reply published through ilo marks the matching inbox item as replied when it can find it locally.

## Keep drafts attached to the right account

```sh
ilo drafts create --account personal --text "Draft text"
ilo drafts list
ilo drafts list --account personal
ilo drafts schedule <draft-id> --at "Friday at 2pm"
ilo drafts publish <draft-id>
```

Draft creation binds the current default or explicit `--account`. Later switches do not change it. `ilo drafts publish <id> --account work` is accepted only when the draft is already bound to `work`; a mismatch fails before a publishing request.

An unassigned draft is allowed when no publishing account exists yet. Scheduling or publishing it binds the account at that point. Schedule values can be ISO timestamps or plain phrases such as `in 30 minutes`, `tomorrow at 9am`, and `Monday at 14:00`.

## Local scheduler

```sh
ilo scheduler run
ilo scheduler watch --interval 60
```

`run` checks once. `watch` keeps running and checks on an interval. Drafts stay in the local SQLite database until they are published. The scheduler uses each draft's bound account, not whichever account is the default when the job runs.

Schedules stay local for both providers. A due Typefully draft is sent with immediate publishing at that time; it does not appear as a future Typefully schedule beforehand.

## JSON output

Commands intended for scripts support `--json`.

```sh
ilo x status --json
ilo accounts list --json
ilo drafts list --json
ilo scheduler run --json
```

## Updates

When a newer npm release is available, an installed global CLI can offer to update itself before a normal interactive command. ilo detects npm, pnpm, Bun, or Yarn and shows the exact command before running it.

The updater never prompts in CI, a non-interactive shell, JSON output, help, version checks, or the MCP server. Set `NO_UPDATE_NOTIFIER=1` to disable the check for interactive commands too.

## Safe automation

Publishing commands are interactive by default. A person sees the final account, provider, text, reply destination, image paths, and alt text before ilo sends anything to a provider. Use `--yes` only in automation that already has an approval step. Monitor refreshes, inbox state changes, reply drafting, follower research, listing, and status checks do not publish anything.

For scripts, prefer JSON output and check the process exit code. Successful commands exit with code `0`; invalid input, missing credentials, and request failures return a non-zero code with an actionable message.

```sh
draft_id=$(ilo drafts create --text "Review this first" --json | jq -r .id)
ilo drafts schedule "$draft_id" --at "2026-07-15T09:00:00+01:00"
```

## Scheduling behaviour

A scheduled draft remains local until a scheduler process finds it due. `ilo scheduler run` performs one check and exits, which suits cron and task runners. `ilo scheduler watch` stays open and is better for a laptop or always-on terminal session.

Natural-language times use your configured local timezone. Use an ISO timestamp with an offset when a post must go out at an exact instant, especially on a server or around daylight-saving changes.

## Files and credentials

ilo keeps configuration under `~/.config/ilo`, local working data in SQLite, and secrets in the system keychain. Back up the database if your draft history matters, but do not copy keychain exports into a repository or shell script.

If a publishing command cannot find the expected account, run `ilo accounts list`. Reconnect Typefully after rotating its key. Use `ilo x connect` after changing direct X app scopes or client configuration.

Publishing setup has no effect on research. Follower and following syncs, article monitors, and reply inbox refreshes fetch public X data independently. Most research commands accept an explicit public handle and do not require Typefully or direct X credentials at all.
