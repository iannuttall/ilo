The `iloso` npm package is the CLI, local MCP server, and TypeScript library. Install it in a Node 22 or newer project when your own tool or agent needs the same article monitoring, reply inbox, follower research, drafting, scheduling, and publishing functions.

```sh
npm install iloso
```

The package is ESM-only and includes TypeScript declarations. There is no separate `@ilo/core` package to install.

## Create and schedule a draft

Import from the package root when your application already knows which action it needs.

```ts
import { createDraft, listDrafts, scheduleDraft } from 'iloso'

const draft = await createDraft('A post to review before publishing')
await scheduleDraft(draft.id, 'tomorrow at 9am')

console.log(listDrafts())
```

Drafts and schedules use the same local SQLite database as the CLI and MCP server. A draft created in code appears in `ilo drafts list` straight away. `createDraft()` is asynchronous because it resolves and snapshots the chosen publishing account before it writes the row.

## Create a reply with images

Pass a reply target and up to four local JPEG, PNG, or WebP files when you create a draft or call `publishPost()`:

```ts
import { createDraft } from 'iloso'

const reply = await createDraft('The result from three companies:', {
  replyToPostId: 'https://x.com/example/status/123',
  images: [
    {
      path: './results.png',
      altText: 'Current follower matches for Cursor, Vercel, and Sentry',
    },
  ],
})
```

ilo normalizes the reply URL to its post id and stores absolute image paths. Keep those files available until a scheduled draft publishes. X applies its own API access and conversation rules when the reply is sent.

Direct X publishing supports the optional alt text shown above. Typefully's documented v2 draft input does not expose image alt text, so a Typefully-bound publish rejects this draft rather than dropping the accessibility text.

## Connect and choose publishing accounts

Most applications should run the interactive `ilo start` flow once. It stores Typefully API keys or direct X OAuth credentials in the operating system keychain, where the library can read them later.

Typefully can also be connected from a trusted setup process:

```ts
import {
  connectTypefully,
  listPublishingAccounts,
  setDefaultPublishingAccount,
} from 'iloso'

const connection = await connectTypefully(process.env.TYPEFULLY_API_KEY ?? '')

console.log(connection.accounts.map(({ alias, username }) => ({
  alias,
  username,
})))

await setDefaultPublishingAccount('work')
console.log(await listPublishingAccounts())
```

`connectTypefully()` checks the key and adds every accessible Typefully social set that has an X profile. The key is saved in the operating system keychain. Do not pass a user-supplied key through logs, query strings, or a hosted ilo service.

Direct X OAuth uses `buildXAuthorizeUrl()`, `exchangeXCode()`, `fetchXMe()`, and `saveXCredentials()` under the hood. The guided CLI owns the callback server and is much less code for a personal integration. Use the lower-level exports only when your application already has a deliberate OAuth UI.

The [publishing accounts guide](/docs/accounts) explains provider choice, local aliases, draft binding, scheduler behaviour, and credential storage.

## Bind work to one X profile

Pass an alias, handle, or full account ID through `account`:

```ts
import { createDraft, publishPost, scheduleDraft } from 'iloso'

const draft = await createDraft('A work account draft', {
  account: 'work',
})

console.log(draft.publishingAccount)

await scheduleDraft(draft.id, 'Friday at 2pm', {
  account: 'work',
})

// Call only after your application confirms this exact action.
await publishPost('A personal post', { account: 'personal' })
```

The saved `publishingAccount` contains the account ID, provider, and X username. Switching the default later does not alter it. `publishDraft(draft.id, { account: 'another-account' })` fails if the selector does not match the bound account.

Pass `account: null` to `createDraft()` when you deliberately want an unassigned local draft. Scheduling or publishing it later binds the selected or default account. Leaving `account` undefined uses the current default when one exists and otherwise keeps the draft unassigned.

## Import and search public followers

The [audience research guide](/docs/audience-research) explains the same workflow across TypeScript, the CLI, and local MCP, including partial coverage, completed snapshots, and refreshes.

Syncs fetch public follower profiles and save their cursor in local SQLite. Limit each call when the account is large, then call it again to resume:

```ts
import {
  getXFollowerProfile,
  getXFollowersStatus,
  searchXFollowers,
  syncAllXFollowers,
  syncXFollowers,
} from 'iloso'

await syncXFollowers({ handle: 'adamwathan', maxPages: 20 })

const status = getXFollowersStatus({ handle: 'adamwathan' })
const result = searchXFollowers({
  handle: 'adamwathan',
  query: 'works at cursor|vercel|sentry',
})

console.log({ status, groups: result.groups })

const profile = getXFollowerProfile({
  handle: 'adamwathan',
  followerHandle: 'leerob',
})

console.log(profile.profile.providerData)
```

Call `syncAllXFollowers({ handle: 'adamwathan' })` when the process should continue until ilo reaches the end of the available list. It checkpoints every page. `getXFollowerProfile()` returns the useful fields plus the complete stored public record without another network request.

The search result includes `coverage.complete`, the number of searchable profiles, X's reported follower total, and the last sync error when one exists. Each group includes conservative current, former, and unclear counts. The `results` array contains current matches and their public bio evidence by default. Pass `includeFormer: true` or `includeUnclear: true` to include those profiles. Search uses SQLite FTS5, so concrete names, handles, phrases, and locations work better than abstract semantic questions.

## Search complete following profiles

The [following search guide](/docs/following-search) covers the complete workflow, including CLI commands, MCP tools, snapshot refreshes, freshness, CSV fields, and current limits.

Sync the profiles an account follows, inspect coverage, and search the same local SQLite index:

```ts
import {
  getXFollowingProfile,
  getXFollowingStatus,
  searchXFollowing,
  syncAllXFollowing,
  syncXFollowing,
} from 'iloso'

await syncXFollowing({ handle: 'adamwathan', maxPages: 20 })

const status = getXFollowingStatus({ handle: 'adamwathan' })
const search = searchXFollowing({
  handle: 'adamwathan',
  query: 'building browser tools',
})

console.log({ status, totalMatches: search.totalMatches })

if (search.results[0]) {
  const stored = getXFollowingProfile({
    handle: 'adamwathan',
    followedHandle: search.results[0].handle,
  })
  console.log(stored.profile.providerData)
}
```

`syncXFollowing()` saves its cursor and resumes an unfinished import. Calling it after a completed import starts a fresh snapshot. Use `syncAllXFollowing()` when a foreground process should continue until ilo reaches the end of the available list, or until the saved profiles match X's reported total and repeated pages add nothing new.

Search reads names, handles, bios, locations, and aliases with SQLite FTS5. It returns every matching profile unless `resultLimit` is supplied. Its `coverage` includes completion, `completionReason`, unique searchable profiles, expected following count, timestamps, last error, age, and a stale flag that becomes true after 24 hours.

`getXFollowingProfile()` returns the saved structured fields and complete public record without another network request. All following functions accept `databasePath` when an application needs an isolated index. `getXFollowingStatus()`, `searchXFollowing()`, and `getXFollowingProfile()` also accept `now` and `staleAfterMs` for deterministic tests or a different freshness policy.

## Monitor and search X articles

Save the handles you want to follow, refresh their article timelines, and search the stored full text:

```ts
import {
  createXArticleMonitor,
  getXArticle,
  listXArticleMonitors,
  refreshXArticles,
  searchXArticles,
} from 'iloso'

createXArticleMonitor({
  accountHandle: 'ilodotso',
  sourceHandle: 'swyx',
})

await refreshXArticles({ accountHandle: 'ilodotso', maxPages: 3 })

const monitors = listXArticleMonitors({ accountHandle: 'ilodotso' })
console.log(monitors.map(({ sourceHandle, historyComplete }) => ({
  sourceHandle,
  historyComplete,
})))

const results = searchXArticles({
  accountHandle: 'ilodotso',
  query: 'browser tools',
})

if (results[0]) {
  const article = getXArticle({
    accountHandle: 'ilodotso',
    identifier: results[0].postId,
  })
  console.log(article.bodyText)
}
```

Refreshes run on demand and save their progress in local SQLite. If `historyComplete` is false on a monitor, call the refresh again to continue from its cursor. Search results contain a short evidence excerpt and body character count. `getXArticle()` returns the complete stored body and public article record.

The [article monitoring guide](/docs/articles) covers cursor state, CLI commands, MCP tools, stored fields, and the limits of the public article timeline.

## Build a local reply inbox

Save X advanced-search queries as monitors and refresh them when your application wants new posts:

```ts
import {
  createXMonitor,
  listXInbox,
  recordXInboxFeedback,
  refreshXMonitor,
  syncAllXFollowing,
  updateXInboxItem,
} from 'iloso'

const monitor = createXMonitor({
  accountHandle: 'ilodotso',
  name: 'ilo mentions',
  query: '"ilo" OR "ilo.so" -is:retweet',
})

await refreshXMonitor({ id: monitor.id, maxPages: 3 })
await syncAllXFollowing({ handle: 'ilodotso' })

const items = listXInbox({
  accountHandle: 'ilodotso',
  status: 'unread',
  iFollow: true,
  language: 'en',
  sort: 'signal',
  explain: true,
  limit: 20,
})

if (items[0]) {
  console.log(items[0].signal)
  recordXInboxFeedback({
    accountHandle: 'ilodotso',
    postId: items[0].postId,
    value: 'useful',
    reason: 'actionable',
  })
  updateXInboxItem({
    accountHandle: 'ilodotso',
    postId: items[0].postId,
    action: 'read',
  })
}
```

Refreshes are on demand and never publish. Saved items include public post text, engagement counts, author fields, monitor evidence, local read/archive/replied state, and the complete public post record. `listXInbox()` can search saved post and author text through local SQLite FTS5. Signal sorting adds a deterministic score, confidence, factors, duplicate evidence, model version, and any saved feedback to each returned item.

`recordXInboxFeedback()` saves a useful or not useful choice in local SQLite. Later posts from the same author receive a cautious adjustment. Pass `value: 'clear'` to remove feedback. The score helps order candidates; it does not prove an author is trustworthy or a post is spam.

`followsMe` and `iFollow` are `boolean | null`. A matching imported relationship can return `true` before a sync finishes. A missing relationship only becomes `false` after the relevant follower or following sync completes; otherwise it stays `null`.

Use `getXInboxItem()` to inspect one result. To prepare a response, pass its `postId` to `createDraft()` as `replyToPostId`. Draft creation remains local and does not need publishing confirmation.

## Validate X post text

Use the exported text helpers before saving content from another editor or agent.

```ts
import { countXPostText, validateXPostText } from 'iloso'

const result = validateXPostText('Post text')

console.log({
  weightedLength: countXPostText('Post text'),
  valid: result.ok,
})
```

## Publish only after approval

The library exposes `publishPost()` and `publishDraft()`, but a function call cannot show ilo's interactive confirmation prompt. Put the approval step in your application and pass only the exact text or draft id the user approved.

```ts
import { getDraft, publishDraft } from 'iloso'

const draft = getDraft('approved-draft-id')

console.log({
  text: draft.body,
  replyToPostId: draft.replyToPostId,
  images: draft.images,
  publishingAccount: draft.publishingAccount,
})
// Call publishDraft(draft.id) only after your application records approval.
```

Run `ilo start` once on the machine before publishing. The library reads the same Typefully or direct X credentials from the operating system keychain. An approval screen should include the provider and X username as well as the content.

## Embed the MCP server

Use the MCP subpath when another Node process needs to construct the server itself.

```ts
import { createServer, registerIloTools } from 'iloso/mcp'

const server = createServer()
```

Most applications should call the root functions directly. `createServer()` and `registerIloTools()` are for MCP hosts that attach their own transport or combine ilo with another local tool server.

## Public exports

The root package exports public X fetch helpers, follower and following sync and search functions, article monitors and search, reply inbox actions, publishing-account storage, Typefully setup and client helpers, draft actions, static image helpers, schedule parsing, X text validation, X OAuth helpers, local configuration, and storage paths. These exports are covered by the packed-package tests, but ilo is still pre-1.0. Pin the package version when another application depends on the exact function signatures.

Use the [CLI reference](/docs/cli) for interactive commands or the [local MCP guide](/docs/mcp) to connect an agent without writing integration code.
